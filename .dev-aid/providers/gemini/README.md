# Gemini CLI Integration for Dev-AID

This directory provides Dev-AID integration for [Gemini CLI](https://github.com/google-gemini/gemini-cli), the secondary supported provider.

## Overview

Gemini CLI uses TOML-based slash commands with `[metadata]` and `[prompt]` sections. Dev-AID provides **31 slash commands** across 5 categories, all with short `aid-*` aliases. Skills are shared via symlinks for automatic synchronization.

## Prerequisites

1. **Gemini CLI installed**
   ```bash
   npm install -g @anthropic-ai/gemini-cli
   # or follow https://github.com/google-gemini/gemini-cli
   ```

2. **Google API key configured**
   ```bash
   export GOOGLE_API_KEY="..."
   ```

3. **Dev-AID installed in your project**
   ```bash
   ls -la .dev-aid/
   ```

## Quick Setup

### Option 1: Symlink-Based (Recommended)

Create symlinks from your project root to use Dev-AID's Gemini configuration:

```bash
# From your project root
ln -s .dev-aid/providers/gemini/.gemini .gemini

# Create GEMINI.md symlink (or copy)
ln -s .dev-aid/providers/gemini/GEMINI.md GEMINI.md
```

### Option 2: Copy-Based

Copy the Gemini configuration to your project:

```bash
cp -r .dev-aid/providers/gemini/.gemini .gemini
cp .dev-aid/providers/gemini/GEMINI.md GEMINI.md
```

## Slash Commands

Gemini uses TOML format for commands (vs Markdown for Claude):

```toml
name = "dev-aid-agent-pr-review"
description = "AI-powered pull request review"

[metadata]
category = "agents"
author = "Dev-AID Team (https://probably.group)"
version = "1.0.0"

[prompt]
content = """
# PR Review Agent

[Full instructions with argument parsing]
"""
```

### Dual Naming System

Same pattern as Claude — every command has a full name and a short alias:

- **Full**: `/dev-aid-agent-pr-review <PR_NUMBER>`
- **Short**: `/aid-pr <PR_NUMBER>`

Type `/aid-` to browse all available commands via autocomplete.

## Hooks Configuration

Gemini CLI hooks are configured via `hooks.toml`:

| Hook | Event | Purpose |
|------|-------|---------|
| `session-start.sh` | Session start | Detect project context and auto-load relevant skills |

The session-start hook:
1. Runs `detect-context.sh` to identify project languages and frameworks
2. Runs `select-skills.sh` to choose relevant skills
3. Loads selected skills into the session context

## Context Loading

Gemini CLI loads context files in this order:

1. `~/.gemini/GEMINI.md` (user's global instructions)
2. Project root `GEMINI.md` (project-specific instructions)

The `GEMINI.md` file is generated by the install script and contains project-specific context.

## Verification

### Check symlinks are working

```bash
ls -la .gemini/skills/
# Should show symlinks to ../../../skills/{core,expert,process}
```

### Check commands are discoverable

In a Gemini CLI session:
```
/aid-help      # Lists all available commands
/aid-status    # Shows configuration and providers
```

### Test a command

```
/aid-onboard   # Should generate an onboarding guide for the project
```

## Troubleshooting

### "Skill file not found"

Verify symlinks are intact:
```bash
ls -la .gemini/skills/expert/fastapi-expert/
```

If broken, recreate:
```bash
cd .gemini/skills
rm -f core expert process
ln -s ../../../../.dev-aid/skills/core core
ln -s ../../../../.dev-aid/skills/expert expert
ln -s ../../../../.dev-aid/skills/process process
```

### "Command not found"

1. Verify `.gemini/commands/` directory exists and contains `.toml` files
2. Check that the symlink from project root points to the provider directory
3. Restart Gemini CLI session to reload commands

### "Hook not executing"

1. Verify hook is executable: `chmod +x .gemini/hooks/*.sh`
2. Check `hooks.toml` has the hook configured
3. Run manually to check for errors: `.gemini/hooks/session-start.sh`
